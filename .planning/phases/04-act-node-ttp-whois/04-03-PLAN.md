<!-- Pattern verificado experimentalmente em RESEARCH.md "Implementation Patterns" (Pattern 1, Pattern 3, Pattern 8): -->

RealSubprocess.run usa `tokio::select! { biased; ... }` com 3 branches:
- cancel.cancelled() -> start_kill + wait + return Cancelled
- tokio::time::sleep(timeout) -> start_kill + wait + return SubprocessTimeout
- child.wait_with_output() -> Ok(SubprocessOutput)
Razao do `biased`: Pitfall 4 (cancel deve ganhar timeout); razao do `start_kill()+wait()` em vez de `kill().await`: Pitfall 4 (evita "OS error 6").

WhoisTtp::execute fluxo:
1. Lock self.last_call (Mutex<Option<Instant>>), calcula `wait = last.and_then(|t| Duration::from_secs(1).checked_sub(t.elapsed()))`, dropa lock.
2. Se wait Some(d), `tokio::time::sleep(d).await`.
3. Invoke subprocess.run("whois", &["-v", "-nobanner", "-accepteula", target], self.default_timeout(), cancel.clone()).
4. Update last_call = Some(Instant::now()) — ANTES do parse (Pitfall 4 corolario: garantir rate limit em retries).
5. Parse via parse_whois_output(&out.stdout). Retorna TtpOutput::Whois(...).
